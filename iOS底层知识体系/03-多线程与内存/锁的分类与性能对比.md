## 1. 锁的分类与性能对比

### 1.1 锁的基本概念

**为什么需要锁？**

多线程编程中，当多个线程同时访问共享资源时，可能会导致数据竞争（Data Race）。锁的作用是保证同一时刻只有一个线程能访问临界区。

**锁的分类**：

| 类型 | 说明 | 实现方式 |
|------|------|----------|
| **互斥锁** | 获取锁失败时会休眠，等待唤醒 | pthread_mutex、NSLock、@synchronized |
| **自旋锁** | 获取锁失败时会忙等待（循环重试） | OSSpinLock、os_unfair_lock |
| **读写锁** | 允许多个读者或一个写者 | pthread_rwlock |
| **条件锁** | 满足特定条件时获取锁 | NSCondition、NSConditionLock |
| **信号量** | 控制并发访问数量 | dispatch_semaphore |

### 1.2 iOS 中常见的锁

#### 1.2.1 OSSpinLock（已废弃）

**特点**：

- 自旋锁，性能最高
- 存在**优先级反转问题**（Priority Inversion）
- iOS 10 后被废弃

**优先级反转问题**：

```
场景：
- 低优先级线程 L 持有锁
- 高优先级线程 H 等待锁
- 中优先级线程 M 抢占 L 的 CPU 时间

结果：H 无法执行，L 无法释放锁，系统性能下降

示例：
线程 1（低优先级）：获取锁 → 被中断
线程 2（中优先级）：占用 CPU，线程 1 无法运行
线程 3（高优先级）：等待锁，但锁被线程 1 持有
→ 线程 3 永远等待
```

**代码示例**：

```objective-c
// ❌ iOS 10+ 已废弃，不推荐使用
#import <libkern/OSAtomic.h>

OSSpinLock lock = OS_SPINLOCK_INIT;
OSSpinLockLock(&lock);
// 临界区
OSSpinLockUnlock(&lock);
```

#### 1.2.2 os_unfair_lock（推荐）

**特点**：

- iOS 10+ 引入，替代 OSSpinLock
- 互斥锁（非自旋）
- 无优先级反转问题
- 性能仅次于 OSSpinLock

**底层实现**：

- 基于 futex（Fast Userspace Mutex）
- 获取锁失败时，线程会休眠（而非忙等待）

**代码示例**：

```objective-c
#import <os/lock.h>

os_unfair_lock lock = OS_UNFAIR_LOCK_INIT;

// 加锁
os_unfair_lock_lock(&lock);
// 临界区
os_unfair_lock_unlock(&lock);

// 尝试加锁（不阻塞）
if (os_unfair_lock_trylock(&lock)) {
    // 加锁成功
    os_unfair_lock_unlock(&lock);
}
```

#### 1.2.3 pthread_mutex

**特点**：

- 跨平台，C 语言接口
- 支持多种类型：普通、递归、错误检查
- 性能中等

**类型**：

```objective-c
// PTHREAD_MUTEX_NORMAL：普通锁（同一线程重复加锁会死锁）
// PTHREAD_MUTEX_RECURSIVE：递归锁（同一线程可多次加锁）
// PTHREAD_MUTEX_ERRORCHECK：错误检查锁（重复加锁返回错误）

pthread_mutexattr_t attr;
pthread_mutexattr_init(&attr);
pthread_mutexattr_settype(&attr, PTHREAD_MUTEX_RECURSIVE);

pthread_mutex_t mutex;
pthread_mutex_init(&mutex, &attr);

pthread_mutex_lock(&mutex);
// 临界区
pthread_mutex_unlock(&mutex);

pthread_mutex_destroy(&mutex);
```

#### 1.2.4 NSLock & NSRecursiveLock

**NSLock**：

```objective-c
@interface NSLock : NSObject <NSLocking>
- (BOOL)tryLock; // 尝试加锁，不阻塞
- (BOOL)lockBeforeDate:(NSDate *)limit; // 超时加锁
@end

NSLock *lock = [[NSLock alloc] init];
[lock lock];
// 临界区
[lock unlock];
```

**NSRecursiveLock**（递归锁）：

```objective-c
NSRecursiveLock *lock = [[NSRecursiveLock alloc] init];

- (void)recursiveMethod:(int)count {
    [lock lock];
    if (count > 0) {
        [self recursiveMethod:count - 1]; // 同一线程可重复加锁
    }
    [lock unlock];
}
```

#### 1.2.5 NSCondition & NSConditionLock

**NSCondition**（条件变量）：

```objective-c
NSCondition *condition = [[NSCondition alloc] init];

// 线程 1：等待条件
[condition lock];
while (!conditionIsMet) {
    [condition wait]; // 释放锁并等待
}
[condition unlock];

// 线程 2：设置条件并唤醒
[condition lock];
conditionIsMet = YES;
[condition signal]; // 唤醒一个等待线程
// 或 [condition broadcast]; // 唤醒所有等待线程
[condition unlock];
```

**NSConditionLock**（条件锁）：

```objective-c
NSConditionLock *lock = [[NSConditionLock alloc] initWithCondition:0];

// 线程 1：等待条件为 42
[lock lockWhenCondition:42];
// 临界区
[lock unlock];

// 线程 2：设置条件为 42
[lock lock];
[lock unlockWithCondition:42];
```

#### 1.2.6 @synchronized

**特点**：

- 使用简单，但性能最差
- 递归锁
- 底层使用哈希表存储锁对象

**底层实现**：

```objective-c
// 编译器转换
@synchronized(obj) {
    // 临界区
}

// 等价于
objc_sync_enter(obj);
// 临界区
objc_sync_exit(obj);
```

**底层原理**：

```
1. objc_sync_enter(obj):
   - 以 obj 为 key，在全局哈希表中查找 SyncData
   - 如果找到，递归计数 +1
   - 如果未找到，创建新的 SyncData，递归计数 = 1

2. objc_sync_exit(obj):
   - 递归计数 -1
   - 如果递归计数为 0，释放锁
```

### 1.3 dispatch_semaphore（信号量）

**特点**：

- 可以控制并发数量
- 也可以作为锁使用

**基本用法**：

```objective-c
// 创建信号量（初始值为 1，作为互斥锁使用）
dispatch_semaphore_t semaphore = dispatch_semaphore_create(1);

// 等待信号（减 1，如果为 0 则阻塞）
dispatch_semaphore_wait(semaphore, DISPATCH_TIME_FOREVER);
// 临界区
// 发送信号（加 1，唤醒等待线程）
dispatch_semaphore_signal(semaphore);
```

**控制并发数量**：

```objective-c
// 最多允许 3 个并发线程
dispatch_semaphore_t semaphore = dispatch_semaphore_create(3);

for (int i = 0; i < 10; i++) {
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        dispatch_semaphore_wait(semaphore, DISPATCH_TIME_FOREVER);

        // 执行任务（最多 3 个并发）
        NSLog(@"Task %d", i);
        [NSThread sleepForTimeInterval:1.0];

        dispatch_semaphore_signal(semaphore);
    });
}
```

### 1.4 锁的性能对比

**性能测试**（10 万次加锁/解锁）：

| 锁类型 | 耗时（ns） | 相对性能 |
|--------|-----------|----------|
| OSSpinLock | 33.5 | ⭐⭐⭐⭐⭐ |
| os_unfair_lock | 43.7 | ⭐⭐⭐⭐ |
| pthread_mutex | 278.1 | ⭐⭐⭐ |
| NSLock | 387.7 | ⭐⭐ |
| NSCondition | 491.1 | ⭐⭐ |
| @synchronized | 770.2 | ⭐ |

**选择建议**：

```
1. iOS 10+：优先使用 os_unfair_lock
2. 需要递归锁：NSRecursiveLock 或 pthread_mutex(RECURSIVE)
3. 需要条件变量：NSCondition 或 NSConditionLock
4. 简单场景：@synchronized（性能要求不高时）
5. 控制并发数量：dispatch_semaphore
```

### 1.5 无锁编程

**原子操作**：

```objective-c
#import <libkern/OSAtomic.h>

// 32 位整数原子操作
int32_t value = 0;
OSAtomicIncrement32(&value);   // value++
OSAtomicDecrement32(&value);   // value--
OSAtomicAdd32(10, &value);     // value += 10
OSAtomicCompareAndSwapInt(0, 1, &value); // CAS（比较并交换）
```

**Swift 原子操作**：

```swift
import Foundation

var value = 0
DispatchQueue.global().async {
    OSAtomicAdd32(1, &value) // 原子加 1
}
```

**适用场景**：

- 计数器
- 简单的状态切换
- 无需复杂临界区的场景
