# 设计一个图片加载框架

> 一句话总结：**图片框架 = 下载（去重+并发） + 二级缓存（内存 decoded + 磁盘 raw/data）+ 解码策略（异步、降采样）+ 与 UI 生命周期绑定（取消、清除）。**

---

## 1. 需求澄清（面试先问）

| 维度 | 选项 |
|------|------|
| 数据源 | 远程 URL / 本地 path / Asset |
| 缓存 | 内存上限、磁盘上限、淘汰策略（LRU）、是否存 decoded |
| 解码 | 是否默认降采样、是否去掉 alpha |
| 线程 | 下载并发度、IO 串行队列、主线程只回读 |
| 生命周期 | UIView 复用时取消旧请求、相同 URL 合并 |

---

## 2. 架构设计

### 2.1 模块划分

```
API 层（UIImageView Category / SwiftUI Wrapper）
    ↓
调度层 Manager（合并请求、选择缓存/下载）
    ↙         ↘
Downloader          Cache
(URLSession)    Memory(NSCache) + Disk(文件+元数据)
```

### 2.2 内存缓存

- **`NSCache`**：线程安全，系统压力下可清；可设置 `countLimit` / `totalCostLimit`（decoded 图用 `cost = 宽高*4` 近似）。
- 存 **UIImage（已解码）** 还是 **Data**：前者命中快占内存大，后者省内存但命中后要解码。

### 2.3 磁盘缓存

- 目录：`Library/Caches` 下独立子目录。
- Key：`URL` 规范化后做 **MD5/SHA256** 文件名，避免非法字符。
- 淘汰：**maxAge**（访问时间 / mtime）+ **maxSize**（超阈值删最旧直到降到 50% 等策略，见旧笔记 SDWebImage 分析）。
- 读写走 **串行队列**，避免文件竞争；大块读写加 `@autoreleasepool`。

### 2.4 下载

- **`URLSession`**，`maxConcurrentOperationCount` 或队列 + 信号量限流（如 6）。
- **相同 URL 合并**：`NSMutableDictionary<URL, [Completion]>` + `barrier` 保字典线程安全。
- **304**：尊重缓存头，可 cancel 并用磁盘数据。
- 后台：**`beginBackgroundTask`** 收尾（参考 SD 笔记）。

### 2.5 解码

- 下载/读盘后在**后台**解码；大图走 **ImageIO 降采样**（见《图片优化》）。
- JPEG 可分质量再压一层（视业务）。

---

## 3. 边界与测试

- **取消**：`dealloc` / `prepareForReuse` 时 cancel `URLSessionTask` 与合并队列中的 token。
- **线程**：回调统一 `dispatch_async(main)` 赋值 image。
- **错误**：失败占位、重试次数、黑名单 URL 防止打爆。

---

## 4. 面试高频追问

- **为什么磁盘命中后还要写内存？** 下次更快；权衡内存压力用 cost limit。  
- **如何避免滚动手势下载风暴？** prefetch + cancel + 合并 + 限并发。
