# 设计一个路由框架

> 一句话总结：**路由把「页面/能力」抽象为可解析的标识（URL 或 路由表项）+ 参数载荷 + 异步结果；核心权衡是可读参数协议 vs 动态编排能力。**

---

## 1. 需求澄清

- **页面跳转** vs **无 UI 能力调用**（如打开某个 SDK、上报）是否统一？
- 参数类型：仅基础类型，还是允许注入对象？（对象通常用 **context / 内存中转池**）
- 是否支持 **Server 下发** path？需要白名单与版本协商。
- **Flutter / RN**：是否需要统一 `Navigator` 之外的 bridge。

---

## 2. 方案对比（旧笔记《知识小集01》）

### 2.1 URL Router

- **形式**：`myapp://module/detail?id=1`
- **优点**：跨端统一、易配置、可服务端驱动。
- **缺点**：参数结构松散、需维护文档；不适合纯工具型模块。

**实现要点**：启动/懒加载时 `register(path, handler)`，`open(url, params)` 查表 dispatch；线程回到主线程做 UI。

### 2.2 Target-Action / 中介者（如 CTMediator）

- **形式**：`performTarget:action:params:`
- **优点**：无需预先注册表，靠约定 + 动态调用。
- **缺点**：字符串 target/action **编译期不检查**，约定成本大。

### 2.3 协议 + 依赖注入

- 适合**服务发现**型能力（账号、登录），不一定走 URL。

---

## 3. 推荐落地结构（混合）

```text
Router.open(Route("goods/detail", params))
    → RouteRegistry 匹配 pattern
    → Builder 构造 UIViewController / Coordinator
    → Presenter push/present
```

- **Pattern**：支持 path 模板与 query。
- **Interceptor**：登录拦截、AB、风控；链式 `next()`。
- **Callback**：`open` 支持 `completion` / `delegate`，跨页面回传用 **闭包中转** 或 **Notification/Combine**。

---

## 4. 面试题

- **路由与组件化关系？** 路由是组件**解耦通信**的一种；还可配合壳工程、动态 framework。  
- **URL 路由如何传 UIImage？** URL 只带 id，大对象放 **内存缓存** 再 key 传递。
