# 行为面试：团队协作与分歧

> 覆盖题干：**与产品经理/设计师如何协作？** **与同事意见不合怎么办？** **跨团队推进怎么做？**  
> 考察点：**沟通成本意识、专业边界、数据与共识、团队目标优先**。

---

## 1. 与 PM（产品经理）

**可讲结构**：

- **需求澄清**：把「要做」翻译成**验收标准**（接口字段、边界、埋点、异常态）。
- **可行性分层**：MVP vs 完整方案；**时间-范围-质量**三角里你给出的选项表。
- **风险前置**：合规、审核、性能、技术债；何时必须拉法务/安全。

**好例子特征**：你主动提出过**替代方案**（例如用现有组件 + 配置达成 80% 体验）。

---

## 2. 与设计师

**可讲结构**：

- **早期介入**：评审阶段指出实现成本与可达性（动效帧率、暗黑模式、大屏适配）。
- **对齐工具**：Figma/Zeplin；标注与切图规范；动效用 Lottie/SwiftUI 等的技术映射。
- **折中路径**：保留视觉重点、简化非关键动效；用**原型或录屏**对齐预期。

**忌**：只说「做不了」；要说「做不了 A，但 B 能接近且成本低 X」。

---

## 3. 与同事技术分歧

**推荐叙事**（STAR 缩短版）：

1. **分歧点**：架构选型 / 接口设计 / 是否引入某依赖——**客观描述**双方关切。
2. **你的动作**：先听完对方约束 → 把分歧写成**可测标准**（性能、可维护性、上线日期）。
3. **解决**：POC、小范围灰度、对照数据；**谁的数据好采纳谁**；若平手则升级决策人。
4. **收尾**：强调**团队采用了什么**，你从对方方案里学到什么。

**红线**：不人身攻击；不把「赢了争论」当亮点，把「团队拿到更好结果」当亮点。

---

## 4. 跨团队（客户端 + 后端 + 数据）

- **接口契约**：版本、错误码、幂等、超时；谁兼容谁。
- **联调节奏**：mock、沙箱、特征开关；问题单归属与 SLA。
- **文档**：一页纸 RFC 或 ADR，避免口头扯皮无据可查。

## 5. 填空模板

```
【场景】与 ________ 在什么环节分歧/协作紧张

【数据或事实】用了 ________ 说服或验证
【结果】上线/范围调整/技术方案：________
【复盘】下次会提前 ________
```

---

## 6. 追问预案

- 「**若对方职级更高仍坚持错误方案？**」→ 书面留痕风险 + 寻求决策人 + 准备降级方案。
- 「**deadline 铁定来不及？**」→ 范围裁剪顺序、并行人力、质量底线（哪些不能砍）。
