## 素材 A：语音房大型重构（架构 + 协作 + 演进）

### 适合回答的题干

「最难的项目？」「做过最大的重构？」「怎么让多人协作不互相踩？」

### 核心矛盾（1分钟版）

- **背景**：业务模块多（RTM/RTC、公屏、麦位、礼物、运营活动等），逻辑堆在 VC，**单文件 3000+ 行 + 分类**，MVVM 也挡不住「改一处要懂全局」；扩展新功能常 **连锁改多处**，多人 commit 冲突多。
- **目标**：模块 **可读、可扩展、可插拔**，且多房型（主题房/高级房等）能 **组合能力**；公共数据 **单一事实来源**，避免插件乱改。

### 技术要点（面试官深挖时展开）

| 点 | 可说的话 |
|----|-----------|
| 形态 | **插件化 + 插件内 MVVM**；插件 = UI + VM + Model，可纯功能型（消息发送、SDK 适配）或容器型（底部 tools） |
| 生命周期 | **PageLife**：主 VC 一条生命周期 **广播** 到各插件（协议对齐 `viewDidLoad` 等）；**Manager + Set** 管理插件；可扩展 **优先级**（同层级 add顺序、先执行高优先级） |
| 上下文 | **Context 协议** 由主 VC 实现：统一暴露 view、房间基础信息等；插件 **只通过 context 取公共数据**，防止随意拿全局单例改状态 |
| 插件通信 | 类 **Swinject** 的 **Container**：`ServiceKey`（协议类型 + name）注册/解析；插件 `register` 对外协议，`resolve` 拿依赖 |
| 分层 | **SDK 层**（声网 RTM/RTC）→ **基础服务**（频道、房间信息、消息处理）→ **业务应用**（礼物、身份卡、公屏等） |
| 消息模型 | `AudioLiveMessage` **双层 struct**：大类分 **频道消息 / UI 消息 / 非 UI 消息**；UI 消息带 sender、装扮等，便于扩展 |
| 消息分发 | **ObserverCenter**：`ObjectIdentifier` + **weak 包装**，避免循环引用；消息插件解析 RTM 后 **遍历派发**；可在此做 **打点、限频、优先级** |
| 公屏映射 | **策略模式**：`MessageUIItemProtocol` 绑定 `showType` ↔ `Cell` 类型；**展示样式与业务消息类型多对多**；Cell 从 UIItem 取文案与点击 |

### 结果与复盘（须自己填数）

- 例如：**需求交付周期**、**冲突次数**、**语音房相关 bug 率** 前后对比；若仍在演进，诚实说「二期在做 xxx」。

### 高频追问

- **「插件会不会越来越像微内核，调试困难？」** → 统一日志前缀、插件清单页（Debug）、单测覆盖 VM；**接口协议化** 保证可 mock。
- **「和组件化、模块化什么关系？」** → Pod 级拆分是 **工程边界**；插件是 **运行时组合**，解决 **同一壳内多房型差异**。

## 素材 B：崩溃治理与 APM（原理 + 落地）

### 适合回答的题干

「最难的技术问题？」「怎么做稳定性？」「堆栈怎么来的、怎么符号化？」

### 分层认知（展示深度）

| 层级 | 代表 | 备注 |
|------|------|------|
| Mach Exception | `EXC_BAD_ACCESS` 等 | 偏底层；可 `task_set_exception_port` |
| BSD Signal | `SIGSEGV` 等 | 多由 Mach 转化；handler 内 **异步信号安全** |
| NSException | `NSRangeException` 等 | 可 `NSSetUncaughtExceptionHandler`；信息较丰富 |
| 难捕获 | Watchdog、部分 OOM/Jetsam | 需 **FOOM/MetricKit/内存** 另案 |

**Swift**：多数走 **Signal 层**，不是 OC 的 NSException 路径（可一句话带过）。

### KSCrash 类方案在答什么（避免只会说「接了个 SDK」）

- **Mach**：异常端口 → monitor 线程 → `thread_get_state` → **unwind/backtrace** → JSON。
- **Signal**：`signal()` 注册；handler 里 **少做事**，记录寄存器/ucontext。
- **堆栈**：**PC/LR/FP**；ARM64 **FP 链**回溯；**全线程 dump**；上报 **原始地址**，**符号化在服务端**（image base + offset → dSYM DWARF）。

### 符号化（追问「offset 怎么算」）

- `instruction_addr - image_base = offset`；在 dSYM **DWARF** 里匹配 **函数区间 + line table**。

### 野指针 / UAF（你文档里的高光）

- **线下**：Malloc Scribble、Zombie、规范（weak self、Timer、KVO、通知移除、Task 取消）。
- **线上**：精确栈往往 **不可求**；**降发生概率 + 大盘下降**；可参考 **SafeObj 思路**：hook retain/release/dealloc 记 **释放档案（环形缓冲）**，崩溃时 **地址与近期释放对象匹配** → **辅助归因**（非100%）。

### 特殊案例（可讲30 秒）

-越狱环境 **hook 框架** + 广告 SDK 队列 → **野指针到无效地址**；处置：**升级 SDK、越狱检测、风控**；说明 **并非都能在自己代码行定位**。

## 素材 C：消息列表卡顿（观测 + 优化闭环）

### 适合回答的题干

「怎么优化列表？」「消息突发怎么不卡？」「Diffable 为什么用？」

### 观测手段

- **线下**：Time Profiler + Core Animation + Allocations；**Mark** 前后对比；压力回放 **真实消息序列**。
- **线上**：FPS（`CADisplayLink` 计数）；**RunLoop + 信号量** 卡顿时长（`beforeSources` / `afterWaiting` 超时）；注意 **功耗与误报**。

### 优化逻辑（与 MessageKit 思想对齐）

- **布局与展示解耦**；富文本 **后台 TextKit（Storage/LayoutManager/Container）** 预排版；**高度与 glyph 提前算好**。
- **自动布局 → 手动 frame**：高频更新下减少 `layoutSubviews` / 文本测量；**position与 size 分开算**。
- **图片**：downsample、**禁止主线程解码**（SDWebImage：`scaleDownLargeImages`、`avoidAutoSetImage` 等，按你实际配置说）。
- **气泡**：resizable image、layer-backed；**静态复杂气泡** 可 **rasterize 缓存**（说清 **动画/分辨率变化下可能失效**）。
- **列表**：`UICollectionView` + **DiffableDataSource**；**增量 snapshot append**，避免全表 diff；突发流量 **流控**（缓冲队列 + 定时批量上屏，如每 0.5s 5 条 + 快进/暂停模式）。

### 结果

- 原 PDF：**FPS 基本50+**；你需换成 **P95帧耗时 / 卡顿率** 等你司真实指标。

## 素材 D：内存与 OOM（原理 + 业务动作）

### 适合回答的题干

「怎么做内存优化？」「OOM 怎么查？」

### 原理短答- iOS：**虚拟内存**、**无 Swap**；关注 **phys_footprint**（如 `task_vm_info`）；**Jetsam** 按优先级与占用杀进程；**内存警告** 来自系统压力，不全是本 App。

### 工具链- **Memory Gauge**（瞬时尖峰）；**memgraph + vmmap/leaks/heap**；Allocations **Generation Mark**；MLeaksFinder / FBRetainCycleDetector（场景与局限）。

### 业务向动作（可直接背）

- **大图**：降采样、`imageWithContentsOfFile` vs `imageNamed`、远程大图缓存策略；**CATiledLayer** 超大图。
- **WKWebView**：`webViewWebContentProcessDidTerminate` **reload**。
- **语音房图片**：**独立 SD命名空间**、限制 memory/disk、**退房清理**；Feed 刷新 **清旧图缓存**。
- **案例**：导航栏暗黑切换在 **scroll 回调里反复 setImage** → 暴涨；**标志位只执行一次**；YY 类 `yy_alignment` 未设宽高大 rect → **注释二分 + Memory Gauge**。

## 素材 E：启动优化（阶段 + 任务分级）

### 适合回答的题干

「怎么做启动优化？」「启动任务怎么管理？」

### 阶段

- **Pre-main**：dyld、rebase/binding、ObjC +load、C++ static initializer。
- **Main**：`didFinishLaunching` → 首屏 `viewDidAppear`。

### 手段

- 减 dylib、**+load 减负**、静态/动态权衡、**order file** 降 page fault；**首屏后异步** 非关键 SDK；**P0～P3 任务分级**。

### 你文档中的工程化表述

- **TPBootor + BootTask**：观察者模式拆 **日志 / 广告 / 网络 / 业务配置**；**syncLaunch vs asyncLaunch**；路由与 **Window/RootVC**；**kinfo_proc 等测启动耗时** 上报。

## 素材 F：FOOM 归因（定义 + 误判 + 演进）

### 适合回答的题干

「FOOM 怎么统计？」「和 crash 什么关系？」

### 判定逻辑（对比思路）

- **自研/Gravity 类**：前台、无 crash、无 App/系统升级等（与 **Facebook 方案** 对比：是否含 **用户强杀、exit、后台态**）。
- **误判**：`ApplicationState` 短暂 Active、**BecomeActive 后秒级内退出**；可 **延迟 N 秒再记一次前台启动**（降误判，非根治）。
- **Crash 漏报**：依赖 Firebase 等时，**写盘失败/二次 crash** 可能漏；可讨论 **本地 crashlog 存在即视为上次非正常**（思路级）。
- **Watchdog**：**无法捕获**，可与 **卡顿监控「最后一帧」** 关联归因。

### 结论话术- DAU 不大、FOOM 量大时 **先粗粒度**；内存治理后再 **对齐 Matrix 等更严口径** 做校准。
