# 行为面试：最大的问题 / 最大挑战

> 题干变体：**「遇到过最大的问题？」「压力最大的一次？」「业务或技术失控边缘你怎么处理？」**  
> 与「最难技术题」重叠时可合并素材，但本题更强调：**影响面、不确定性、多方干系人、时间与止损**。

---

## 1. 与「最难解决的问题」怎么分工

| 本题侧重 | 「最难」题侧重 |
|-----------|----------------|
| 范围大、风险高、推进难（人/流程/时间） | 单点深挖、根因难找、长期卡壳 |
| 「怎么摆平局面」 | 「怎么证明根因并修掉」 |
| 可适度技术泛化 | 最好有**可复述的排障链** |

**建议**：准备 **2 个不同故事**，避免两题讲同一个段子。

---

## 2. 推荐叙事骨架（STAR + 风险表）

1. **严重性（30 秒）**：影响多少用户/多少收入/是否阻塞发版；**不要夸张**，面试官会追问验证方式。
2. **干系人**：谁在乎这件事（老板、客户、客服、合规）；你如何**同步信息频率**（日报/群/会议）。
3. **分析与切片**：如何把大问题拆成**可验证假设**（二分法、灰度、对照组）。
4. **并行路径**：主方案 + **Plan B**（降级、开关、回滚包）；体现你不是单线程赌徒。
5. **收尾**：指标恢复、复盘文档、**防止再发**的机制（监控、告警、Checklist）。

---

## 3. 好答案的「硬指标」

至少出现以下**两类**中的两类：

- **时间**：几小时/几天内到什么状态；里程碑。
- **量化**：崩溃率、超时率、客诉量、SLA、亏损避免额（合理即可）。
- **行为**：你**推动**了谁（拉会、写 RFC、要资源）。

---

## 4. 常见坑

- **只抱怨**：前任差、需求乱、测试不给力——面试官会担心你入职后也这样讲他。
- **全是英雄主义**：一个人通宵拯救世界——Senior 岗位会怀疑**协作与可复制性**。
- **没有止损**：听不到「先止血再根治」；成熟工程师会先**控制爆炸半径**。

---

## 5. iOS 场景举例方向（择一深挖）

- **线上崩溃/ANR 飙升**：聚类、符号化、是否与某次发版/系统版本相关；热修/远程开关；与后端联合定位。
- **发版前关键缺陷**：范围裁剪、Feature Flag、强制升级策略的利弊。
- **第三方 SDK 失控**：供应商响应慢时的**临时封装、降级、替换路线图**。
- **性能与成本**：带宽/CDN/客户端缓存策略在活动期间的失控与预案。
- **大型业务重构（如语音房）**：单 VC 数千行、多人冲突、扩展牵一发动全身；**插件化 + 生命周期同步（PageLife）+ Context 单一数据源 + 容器式依赖解析**；分阶段替换与验证——见 [实战素材-语音房与工程难题.md §素材A](./实战素材-语音房与工程难题.md#素材-a语音房大型重构架构--协作--演进)。

## 6. 填空模板

```
【问题类型】稳定性 / 性能 / 安全 / 合规 / 协作阻塞 / 其他
【影响】用户比例或核心路径是否不可用；持续多久
【干系人】我汇报给谁；需要谁配合
【第一反应】止血动作：________（回滚/开关/降级）
【根因定位】如何证明是 ________ 而不是 ________
【主方案】步骤 1→2→3；周期
【Plan B】若主方案失败：________
【结果】指标 + 复盘沉淀（监控/流程/文档）
```

---

## 7. 追问预案

- 「**当时最坏情况是什么？**」→ 诚实描述，并说你为最坏情况留了什么后手。
- 「**老板和客户意见冲突你帮谁？**」→ 先对齐**业务目标与风险承受度**，再给技术选项与成本。
- 「**若资源只有一半？**」→ 砍范围顺序：先保核心链路，延后锦上添花。
- 「**谁该背锅？**」→ 转成**系统改进**：流程上如何避免单点人为失误。
