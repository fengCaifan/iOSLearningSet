# 行为面试：最满意 / 最好的项目

> 题干变体：**「做过最满意的项目？」「最有成就感的事？」「请介绍一个最能代表你水平的项目。」**  
> 考察点：**选型与 ownership、技术深度与取舍、业务结果、复盘能力**——不是项目越大越好，而是**你的贡献可讲清楚、可验证**。

---

## 1. 面试官真正在听什么

- **你是不是核心贡献者**：需求里有多少是你发起/拍板/落地的，而不是「我们团队做了」。
- **你有没有工程判断力**：在约束（时间、人、遗留代码）下做过哪些**取舍**，而不是堆栈罗列。
- **结果是否可信**：尽量用**指标、版本、范围**（例如：冷启动 P90、崩溃率、包体、首屏时间、客诉量）。
- **你能不能自我批评**：哪块若重来会改——显得成熟，不是减分。

---

## 2. 推荐结构（STAR + 一层「Why」）

| 块 | 时长占比 | 内容 |
|----|-----------|------|
| **S** | ~15% | 业务背景、用户/客户是谁、当时痛点（一句话量化） |
| **T** | ~15% | 你的职责边界：你负责哪条链路/模块，成功标准是什么 |
| **A** | ~50% | **3～5 个关键行动**，每个带「决策理由」：为什么选 A 不选 B |
| **R** | ~15% | 指标前后对比；若失败写「验证了什么」 |
| **Why 满意** | ~5% | 与个人成长或价值观挂钩，忌空喊「有挑战」 |

**忌**：五分钟讲完还没出现「我负责…我做了…」；全程只有第三方库名字。

---

## 3. 好故事 vs 坏故事（自检）

| 坏信号 | 好信号 |
|--------|--------|
| 只讲公司多牛、产品多火 | 讲清你在其中的**决策与执行** |
| 全是「我们」，没有「我」 | 每个段落有**第一人称动作** |
| 结果只有「上线成功」 | 有**前后数字**或用户/业务反馈 |
| 技术栈报菜名 | 每个技术点对应**要解决的问题** |
| 没有冲突与取舍 | 至少一个**权衡**（时间/质量/风险） |

---

## 4. iOS 方向可嵌的「硬细节」示例（替换为你真实项目）

在 **Action** 里穿插 1～2 个可追问的技术点即可，例如：

- **启动/包体**：Instruments App Launch、二进制重排、`+load` 审计、动态库懒加载、冷启动阶段拆分。
- **稳定性**：符号化、Key crash 聚类、OOM 与内存图、灰度与回滚策略。
- **性能与体验**：卡顿主线程栈、列表预加载、磁盘与图片缓存策略、弱网。
- **架构与协作**：模块化边界、接口层设计、Feature Flag、与后端/数据约定。

**原则**：每个点都能用一句话说清**问题 → 手段 → 结果**。

## 5. 填空模板（复制到备忘录里写）

```
【项目】名称 / 周期 / 端（iPhone+iPad 等）
【业务】解决谁的问题？（用户/内部运营/B 端客户）
【我的角色】Title；是否 Owner；团队规模；向谁汇报
【成功标准】上线日期？必须满足的指标？（写 2 个数字）

S：背景 + 痛点（1～2 句，带数字）
T：我具体负责的范围 + 约束（人力、遗留、合规等）
A1：________ 为什么这样做 ________
A2：________ 备选方案是什么，为何没选 ________
A3：________ 与谁对齐（PM/设计/后端）________
R：指标 before → after；有无二次迭代
Why 满意：我学到了 ________ 或价值观上是 ________
```

---

## 6. 高频追问与答法方向

1. **「你最大的贡献是什么？」** → 一句话结论 + 一个别人做不了的点（排障、架构、推动跨团队）。
2. **「如果重来一次你会怎么做？」** → 说一个**具体**改进点（监控前置、灰度策略、技术债偿还顺序），不要全盘否定。
3. **「项目里最大的风险？」** → 风险是什么、你如何**量化**、如何**降级**或回滚。
4. **「和意见不合的人怎么合作？」** → 见 [团队协作与分歧.md](./团队协作与分歧.md)，不要人身攻击前同事。
5. **「数据怎么采集的？可信吗？」** → 埋点/日志/AB/Instruments/线上大盘，诚实说局限。

---

## 7. 口述时间控制

- **第一遍**：完整 STAR，**2～2.5 分钟**。
- **被追问后**：每个追问 **30～45 秒**，只答该问，不重复整个故事。
