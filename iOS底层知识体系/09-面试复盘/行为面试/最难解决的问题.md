# 行为面试：最难解决的问题

> 题干变体：**「技术上卡最久的一次？」「最难的 bug？」「做过最复杂的排查？」**  
> 考察点：**问题建模、假设验证、工具链、根因与泛化**——比「最大挑战」更偏**技术深度与思维过程**。

**已按「真实技术向素材」整理的长版提纲**（语音房、崩溃、卡顿、内存、启动、FOOM）见 → [实战素材-语音房与工程难题.md](./实战素材-语音房与工程难题.md)。下面仍保留**通用叙事结构** + **题型索引**。

---

## 0. 「最难」应优先落到具体技术题

面试官问「最难」时，**优先选可深挖的一条技术线**，而不是只讲「项目很大、人很多」。下面几类在 iOS 侧 **最扛追问**（与 [实战素材](./实战素材-语音房与工程难题.md) 一一对应）：

| 类型 | 你应能讲清的「硬核」 |
|------|----------------------|
| **崩溃治理** | Mach / Signal / NSException 分层；KSCrash 类：**异常端口、FP 回溯、全线程、JSON 上报、服务端符号化**；野指针 **线下 Zombie/Scribble** vs **线上辅助归因（如释放档案匹配）** |
| **FOOM** | 判定条件与 **误判**（前后台态、强杀、exit）；与 crash 漏报、Watchdog 的关系 |
| **消息列表卡顿** | RunLoop/信号量卡顿检测；**后台 TextKit**、手动布局、Diffable **增量 snapshot**、**流控队列** |
| **内存/OOM** | `phys_footprint`、Jetsam；**大图 downsample**、独立缓存命名空间与退房清理；Allocations **Generation** vs Memory Gauge **尖峰** |
| **启动** | Pre-main vs Main；**BootTask 分级**、sync/async、order file、+load |

**口径**：OOM/footprint **与机型、系统相关**，对外说「我们以 Instruments / 线上监控 / 业务指标为准」，避免死记单一比例。

## 1. 好素材长什么样

优先选满足多条的故事：

- **表象迷惑**：日志少、复现率低、只在特定机型/系统/网络出现。
- **排查链长**：你用了**多种手段**交叉验证（本地复现、线上日志、二分注释、Instruments、网络抓包等）。
- **根因反直觉**：不是第一眼怀疑的那一行；最后能**一句话说清根因**。
- **有沉淀**：文档、单测、防御代码、监控——证明你不会第二次踩同坑。

**避免**：纯依赖「升级库就好了」且说不清为什么；或保密到无法讲技术细节（可脱敏讲思路）。

## 2. 叙述节奏（建议 2～3 分钟）

1. **现象（15%）**：用户/监控看到什么；为何「难」（偶发、跨模块、三方黑盒）。
2. **初步假设（10%）**：你先怀疑什么；**如何快速证伪**（很重要，体现思维）。
3. **缩小范围（40%）**：二分、对照实验、最小复现 demo、版本/提交 bisect。
4. **根因（20%）**：一句话 + 可选「为什么之前没人发现」。
5. **修复与验证（15%）**：单测/灰度/指标；若 trade-off 说明取舍。

**忌**：从第一秒开始念代码；没有「走弯路」会显得假——可以保留**一条**走弯路的简短提及再拉回。

## 3. iOS 常见「难」题类型（用于定位你的素材）

| 类型 | 可讲的工具/方法 |
|------|------------------|
| 偶现崩溃 | 符号化、僵尸对象、Thread Sanitizer、崩溃聚类、Key path |
| 内存/OOM | Memory Graph Leaks、Allocations、大图片/缓存、循环引用 |
| 卡顿 | Time Profiler、主线程栈、锁竞争、IO 在主线程 |
| 网络 | Charles/Proxyman、弱网模拟、超时与重试幂等 |
| 编译/链接/包体 | ld 参数、静态/动态库、Swift/OC 混编边界 |
| 系统行为差异 | 特定 iOS 版本 API 变更、隐私权限、后台限制 |

## 5. 高频追问

- **「如何保证不再发生？」** → 监控/告警/发布 Checklist/单测覆盖边界。
- **「为什么是你而不是别人找到？」** → 工具熟悉度、耐心、跨模块视野（忌贬低同事）。
- **「修复有没有副作用？」** → 诚实说 trade-off，以及你如何验证副作用可控。
- **「若完全无法复现？」** → 加强日志与采样、版本对比、用户侧信息采集的边界与隐私。

## 6. 与「浅层示例」的升级关系

旧手册用「崩溃率 + 第三方 SDK」一句话带过；**达标回答**需自备：

- **时间线**（发现 → 定位 → 上线补丁的天数/小时）；
- **至少一个证伪步骤**；
- **根因的一句话定义**；
- **工具链名词能对上**（例如：不是只说「接了崩溃 SDK」，而是能说 **堆栈从哪来、符号化靠什么、野指针线上能做什么**）。
