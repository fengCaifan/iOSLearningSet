## 1. 开场自我描述（先说完整故事，再进技术细节）

> 这一段是为了避免你说得太碎。先用 60～90 秒把「我是谁、负责什么、为什么做这件事、做到什么结果」一次性讲完，再根据面试官兴趣展开追问。

### 口播模板（可直接用，替换其中数据）

在我最近几年的项目里，我最有代表性的一个是 **Gravity 语音房的大型重构**。我在这个项目里承担的是核心架构落地和跨模块推进，具体覆盖房间主链路里的消息、公屏、麦位、工具区等模块协作。这个重构的起点是房间页长期演进后出现了典型的工程失控：单 VC 体量过大、多人并行开发互相干扰、新房型和新玩法接入成本越来越高，导致交付速度和线上稳定性都受到影响。  

我当时把目标定得很明确：第一，房间能力要能按房型组合，也就是可插拔；第二，公共状态要有单一事实来源，减少隐式共享状态；第三，在不 Big Bang 推翻重来的前提下分阶段迁移，确保业务持续交付。围绕这个目标，我推动落地了 **插件化 + 插件内 MVVM**，并配套 **PageLife 生命周期分发、Context 读写边界、Container 协议解耦、消息分层与统一派发**。  

最终这个改造带来的价值，不只是代码变“好看”，而是实实在在体现在项目上：协作冲突和回归成本下降、需求接入更可预测、房型扩展更快、线上问题定位边界更清晰。对我来说，这个项目是“最好”的原因在于，它同时解决了 **技术复杂度、团队协作效率、业务迭代速度** 三件事，而且方法可以迁移到别的复杂页面，不是一次性的局部优化。

## 2. 主故事：一题通吃版（约 2～3 分钟口播）

### S（情境）

Gravity 语音房是核心营收场景之一，能力面非常宽：**RTM/RTC、公屏、麦位、成员、礼物、运营活动、小游戏** 等都堆进房间页。历史实现是 **单 VC 3000+ 行 + 多分类**，MVVM 也挡不住「改一处要懂半本书」；新需求经常 **牵一发动全身**，多人同时改同文件，**合入冲突和回归成本** 都偏高。不同 **房型**（如主题房、高级房）需要的业务能力 **组合不同**，用继承堆子类会爆炸。

### T（任务与约束）

目标可以压成三句话：

1. **工程上**：把「房间壳」和「业务能力」拆开，能力 **可插拔、可组合**，不同房型只装配不同插件集。  
2. **数据上**：房间级状态要 **单一事实来源**，插件不能各改各的全局。  
3. **协作上**：让同学能 **相对独立** 在插件内迭代，主壳稳定、减少互踩；上线策略上最好能 **分阶段迁移**，而不是 Big Bang 一次换心。

### A（行动 — 分层次说清「你做了什么」）

1. **形态：插件化 + 插件内 MVVM**  
   每个插件打包自己的 UI / VM / Model；有的是 **纯能力型**（如消息发送、SDK 适配），有的是 **容器型**（如底部工具区）。主 VC 只负责 **装配、生命周期广播、提供 Context**。

2. **生命周期：PageLife**  
   定义 **`TPPageLifeService`**，方法语义对齐 `viewDidLoad` / `viewWillAppear` 等；**PageLifeManager** 用 **Set** 管插件，主 VC 在系统回调里 **统一遍历分发**。需要时加 **优先级**（同层级 add 顺序、先执行高优先级业务），避免「谁先谁后」靠口头约定。

3. **上下文：Context 协议**  
   由主 VC 实现，对外只暴露 **根 view、房间基础 Model、必要容器** 等。插件 **只通过 Context 读公共数据**；写路径收敛到 **房间服务 / 房间级 VM 的 API**，避免随手改单例字段。

4. **插件间通信：类 Swinject 的 Container**  
   **`TPContainerProtocol`**，用 `ServiceKey`（协议类型 + 可选 name）做 **register / resolve**。插件在 `pluginDidLoad` **注册对外协议**；其他插件 **resolve 后直调**——**编译期面向协议**、可单测替实现，比到处 `NSNotification` 字符串更可控。一对多广播仍走 **ObserverCenter**（见下）。

5. **消息：模型分层 + 统一入口派发**  
   `AudioLiveMessage` 一类结构按职责拆：**频道消息 / UI 消息 / 非 UI 消息**（展示责任、是否上公屏、是否跑动画各不同）。**消息处理插件** 解析 RTM 后，经 **ObserverCenter** 派发给订阅方；Center 里用 **`ObjectIdentifier` + weak 包装** 控循环引用。统一入口便于做 **打点、限频、优先级、顺序依赖**。

6. **公屏：策略模式控 Cell 爆炸**  
   `MessageUIItemProtocol` 把 **`MessageShowType` ↔ Cell 类型** 绑在一起；**展示样式与业务消息类型多对多**——新消息类型优先 **复用已有 showType**，减少新 Cell 种类。

7. **协作与工程侧（别忘了说「人」）**  
   分模块迁移、主路径先切壳再迁业务；Code Review 强调 **协议边界**；Debug 可配合 **插件清单/统一日志前缀**（与实战素材中追问一致）。

### R（结果 — 不只写数字，要写“项目贡献”）

建议按下面四层讲结果，这样面试官能听到“重构/插件化为什么值得做”：

1. **工程效率层（研发侧）**  
   - 例如：合入冲突下降、跨模块改动面收敛、平均需求开发周期缩短。  
   - 可讲法：以前一个玩法改动要摸 4~5 个区域，现在多数在单插件内闭环。

2. **质量稳定层（线上侧）**  
   - 例如：语音房相关缺陷率下降、回归问题减少、问题定位路径缩短。  
   - 可讲法：生命周期和消息入口统一后，线上问题能更快定位到“哪个插件、哪条链路”。

3. **业务迭代层（产品侧）**  
   - 例如：新房型或新玩法接入时间变短，试验能力更强。  
   - 可讲法：房型差异由“改主壳 if-else”变成“插件装配差异”，所以扩展成本更可控。

4. **团队协作层（组织侧）**  
   - 例如：并行开发互踩减少，新同学上手路径更清晰。  
   - 可讲法：模块边界和协议边界明确后，职责拆分更稳定，评审与交接成本下降。

### 填空模板（按真实情况替换）

- 迁移前后，语音房相关需求从「平均 `X` 天」缩短到「`Y` 天」。  
- 合入冲突从「每迭代约 `X` 次」降到「`Y` 次」。  
- 线上相关缺陷率/回归率从「`X%`」降到「`Y%`」。  
- 新房型接入由「需要改主壳多个位置」变为「通过插件组合，核心改动集中在 `X` 个插件」。

如果暂时没有完整统计，也可以用阶段性表达：  
「核心链路已迁完，新需求默认走插件；旧链路处于退坡阶段，当前收益主要体现在并行开发冲突和回归成本下降。」

### L（反思 — 面试里很加分，别省）

- **和「组件化 / Pod 模块化」什么关系？**  
  Pod 是 **工程边界、编译期拆分**；插件是 **运行时在同一个房间壳里组合**——解的是 **多房型差异、热插能力** 的问题，两个维度 **互补不互斥**。

- **代价是什么？**  
  间接调用多、排查路径长 → 靠 **统一日志、Debug 插件列表、协议化可 mock、VM 单测** 把调试成本收住；否则插件化会被人诟病「微内核地狱」。

- **若重来会怎么做？**  
  可提一条真实复盘：例如 **更早** 定 Context 的读写规范、**更早** 对消息大类做图式化文档、或某条链路的 **Feature Flag** 可以更早做。

## 3. 为什么这是「我做过最好的项目」

这一段专门回答「最好项目是什么」。不要只说“技术难”，要说“价值最大、可迁移、你的成长最完整”。

### 建议口播（30～45 秒）

我会把 Gravity 语音房重构当作我做过最好的项目。因为它不是单点性能优化或一次性救火，而是把一个高复杂度核心业务，从“人驱动、经验驱动”升级成“边界清晰、结构可演进、团队可并行”的工程体系。它的价值同时落在四个层面：研发效率、线上稳定、业务迭代速度、团队协作成本。对我个人来说，这个项目也让我完成了从“能写功能”到“能设计并推动复杂系统演进”的能力跃迁。

### 可补一句差异化

如果面试官再追问“为什么不是你做过的某次性能优化”，你可以答：性能优化通常是局部收益，而这个重构改变了后续多个季度的开发方式，收益更长期、可复利。

### Q1：当时最难的一点到底是什么？你凭什么判断「必须动架构」而不是继续堆功能？

#### Q1 口播（STAR-L）

- **S/T**：最难的是 **业务与协作双重耦合**——单 VC 承载太多横切能力，需求一多就变成「全局智力游戏」；房型的差异又要求 **树状组合**，继承体系扛不住。  
- **A**：和 TL 对齐 **可量化的痛**（冲突次数、缺陷回归面、排期风险），定目标 **插拔 + 单数据源 + 清晰协作边界**，再落 PageLife/Context/Container。  
- **R/L**：用迁移节奏控制风险；**L**：堆功能只能短期交付，**长期**会拖垮并行开发与线上稳定——所以是 **由 pain 驱动、由指标收口**，不是为架构而架构。

#### Q1 深度（面试官爱追问）

- 说清 **三样东西各解决什么**：**PageLife** 解决与系统生命周期对齐和顺序；**Context** 解决「读从哪来、写往哪去」；**Container** 解决 **跨插件、面向协议** 的定点依赖，**不是** 用来替代所有通知。

#### Q1 防守

若问「何不时序图重构」：答 **房型组合 + 并行团队** 需要运行时装配，比单纯抽方法对象更贴业务。

---

### Q2：PageLife 和直接 swizzle 或子 VC 嵌套比，你咋选？会不会漏调、调两次？

#### Q2 口播（STAR-L）

- **S/T**：要的是 **和 UIKit 同构、可测、不黑魔法** 的生命周期广播。  
- **A**：**主 VC 唯一出口** 调 Manager → 再调各插件；插件 **禁止** 自己再 swizzle 宿主；优先级用于顺序敏感场景。单测可 mock Manager 验证 **调用顺序/次数**。  
- **R/L**：漏调/双调风险集中在 **主 VC 一处**，比「到处监听」好审计；**L**：这是 **可归纳的规约**（Code Review 可拦），不是依赖个人记性的约定。

#### Q2 深度

- 可提「扩展点」：在系统回调之上加 **`pluginDidLoad` 等更早钩子**（若你项目确有）。  
- 若问 **与 Child VC**：Child VC 适合大页面分区；**插件是同一页内细粒度能力**，很多插件 **不持有独立 UIKit 容器层**，用 PageLife 更轻。

---

### Q3：Context 怎么设计才能既不臃肿又不被人绕过？

#### Q3 口播（STAR-L）

- **S/T**：要 **收敛读、收敛写**；不能让插件各自 `shared` 改状态。  
- **A**：Context **协议化**，只放「房间级真的公共」的 getter；**写** 只通过 **房间服务/VM** 暴露的 API。代码规范 + Review 抓 **越权访问**；新需求默认 **新能力进插件、读 Context**。  
- **L**：Context 会 **慢慢变肥** 是真实风险；缓解是 **按域拆协议**（如 `RoomMediaContext` / `RoomIMContext`）或阶段性梳理，别一口气上帝对象。

#### Q3 深度

- 消息 struct **分层**（频道 / UI / 非 UI）与 Context **职责分离** 是同一思想：**在模型层就把「谁负责展示/透传」说清楚**。

---

### Q4：Container 和通知中心、router、URL scheme 各干什么活？

#### Q4 口播（STAR-L）

- **A**：**点对点、面向协议、可替换实现** 用 Container **resolve**；**一对多、领域事件** 用 **ObserverCenter**（你已在消息总线上用）。不鼓励所有事都广播，否则 **调试时找不到边**。  
- **L**：Router/URL 更适合 **跨大模块/远程拉起**；**同进程、同房间壳内** 的插件协作，**协议 + 容器** 更直接、类型更安全。

#### Q4 深度

- 对比 Swinject：思路一致；说清楚 **ServiceKey 为何需要 name**（同协议多实现，如多实例策略）。

---

### Q5：房间消息全链路要多人响应，怎么做到可治理？

#### Q5 口播（STAR-L）

- **S/T**：多模块订阅同一类 **房间事件**，要 **统一治理**（打点、限频、顺序）。  
- **A**：**消息处理插件** 做 RTM → 模型 → **分类** → ObserverCenter 派发；Center **weak 引用** 避免循环。热点路径加 **流控/优先级** 策略（与 [面试模拟-03](./面试模拟-03-消息列表卡顿优化.md) 的列表侧优化 **衔接**：那边管上屏，这里管「进房后事件总线」）。  
- **L**：「统一入口」是 **可运营** 的技术资产，否则每个业务自己加通知会失控。

#### Q5 深度

- 能各说一句：**为何三类消息**（频道 / UI / 非 UI）——**发送方、展示责任、是否占公屏** 不同，混在一起会在 Cell 和 VM 里到处 `if`。

---

### Q6：公屏 Cell 类型爆炸，你具体怎么「收口」？

#### Q6 口播（STAR-L）

- **S/T**：业务消息类型疯长，若 **一类型一 Cell** 维护成本指数上升。  
- **A**：**策略模式**：`MessageUIItemProtocol` 绑定 `MessageShowType` 与 **Cell 复用**；**多对多** 映射。  
- **R/L**：新需求优先 **复用 showType**；真要新样式才加 Cell。  
- **L**：这是 **用「展示维」** 对 **「业务维」** 做降维，和 IM 里「气泡样式枚举」是同一类套路。

#### Q6 深度

- 追问 **Diffable 的关系**：**showType/高度计算** 在 dataSource 前就已稳定，更好做 **稳定 identity** 与 **增量**（与模拟卷 03 互文）。

---

### Q7：分层从口头到工程怎么对齐？

#### Q7 口播（一屏版）

- **SDK 层**：RTC/RTM 等厂商能力封装。  
- **基础服务层**：频道、房间元信息、消息管道。  
- **业务应用层**：礼物、身份卡、公屏等 **插件集**。  
- 面试官若问 **依赖方向**：业务 → 基础服务 → SDK；**禁止** 反向穿透（可在答案里点一句）。

## 7. 高频追问一句答（快表）

| 追问 | 一句答 |
| ------ | -------- |
| 插件化会不会难调试？ | 统一规范：**日志前缀、Debug 清单、协议可 mock、VM 单测**；主 VC 只负责 **组装与分发** 不膨胀业务。 |
