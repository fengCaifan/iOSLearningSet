# 面试模拟 07：语音房、RTC 与「接第三方 SDK」— 理论速记

> **用途**：你有多段「语音房 + 接厂商 RTC/IM SDK」经历，但**未自研编解码/传输栈**时，用本文补齐**面试可问到的概念层**与**叙事话术**，避免只会调接口。  
> **和 §01 的关系**：[面试模拟-01-语音房大型重构.md](./面试模拟-01-语音房大型重构.md) 偏 **iOS 工程与架构**；本文偏 **音视频链路 + 接 SDK 边界**。对外仍可与行为素材 [实战素材-语音房与工程难题.md](../行为面试/实战素材-语音房与工程难题.md) 同主题呼应。

---

## 0. 先有一句「防守型」总述（可背诵）

> 「房间内的采集、编码、弱网对抗、与媒体服务器的传输协议细节，是 **SDK/云服务厂商** 实现；我侧负责在 **iOS 生命周期、业务状态机、与信令/房间服务** 的衔接下，保证 **进房成功率高、首帧/卡顿可观测、问题可复现、与厂商能闭环**。」

**要点**：不伪装读过 RTC 内核；把**价值**落在 **工程落地、指标、排障、协作** 上，同时**概念上能画通一条链路**。

---

## 1. 从「语音房」到「媒体面」要分清的三层

| 层 | 常见职责 | 面试常问点 |
|----|----------|------------|
| **业务与 UI** | 麦位、成员列表、公屏/礼物、房管、活动 | 状态机、多页面生命周期、和 IM 的边界 |
| **信令 / 房间服务** | 进房/退房、上麦/下麦、踢人、房间属性同步 | 与「媒体通道」是两条线：信令通 ≠ 能听见 |
| **媒体（RTC）** | 采集、编码、传输、混音/订阅、播放 | 延迟、弱网、设备切换、3A、首帧 |

**可画图的一句话**：`信令` 决定「**谁在什么角色**」；`媒体` 决定「**声音/画面怎么到**」。很多线上问题是 **信令已成功但媒体未起流**，或 **反过来的注销顺序错误**。

---

## 2. 端到端链路（概念级，够应付大部分追问）

用「语音」一条线能讲清即可（视频多一路画面 + 可能带宽策略不同）：

1. **采集（Capture）**：麦克风 PCM；移动设备上要关心 **系统音频路由**（听筒/外放/蓝牙）、**采音权限**、**他 App 占麦** 等。  
2. **前处理 / 3A（见 §5）**：回声消除 AEC、噪声抑制 ANS、自动增益 AGC。  
3. **编码（Encode）**：常见语音编码如 **Opus**（低延迟、抗丢包好）；**码率/采样率** 和清晰度、流量、发热有关。  
4. **封包 + 传输**：多为 **UDP + 厂商自有或类 RTP 协议**；靠 **JitterBuffer（抖动缓冲）** 把「不匀速到达」的包变成**稳定可播**。  
5. **云端**：混流、转码、转推 CDN（直播混流场景，语音房可能简化）、路由到订阅者。  
6. **接收端**：**JitterBuffer → 解码 → 播放**；若延迟堆积，会听到「越来越滞后」或策略性丢包/加速。

**面试短答模板**：  
「延迟来源大致有：采集缓冲、编解码、网络传播、JitterBuffer 等待、播放缓冲；弱网时还会触发 **FEC/NACK/重传** 等，都是 **时延和流畅度** 的权衡。」

---

## 3. 和 SDK 集成的「客户端职责清单」

不背厂商 API 名，背**类别**即可：

- **身份与鉴权**：Token、AppId、UserId 是否和信令/业务一致；过期与刷新策略。  
- **房间/频道生命周期**：`initEngine` / `enterRoom` / `leave` / `destroy` 的**顺序与幂等**；重复进房、快速切房。  
- **设备与路由**：切蓝牙、来电话、闹钟、Siri、其它 App 抢音频会话（**AVAudioSession** 相关，见 §6）。  
- **前后台与锁屏**：后台是否保活、是否仅音频；**被挂起**后重连。  
- **业务状态与 SDK 的绑定**：上麦再 `publish`、下麦 `unpublish`；**麦位与媒体角色** 必须一致。  
- **可观测性**：进房耗时、首帧、音画不同步、断线重连次数、**厂商回调里的错误码** 与 **自有埋点** 对齐。

---

## 4. 语音房业务名词（和面试官对齐语义）

- **流（Stream/Track）**：一条发送或接收的 **音频/视频轨**；**旁路推直播** 时常见「一条 RTC + 一条转推」等组合。  
- **角色**：主播/嘉宾/观众；是否 **可发流** 由业务 + 信令 + SDK 共同约束。  
- **混流 / 订阅**：是「每人单独拉多路」还是「云混成一路」；影响 **端上 CPU/带宽** 和 **连麦方数量策略**。  
- **信令与 IM**：房态、麦位、公屏、礼物常走 **长连接/IM**；别和 **媒体连接状态** 混为一谈。  
- **1v1 连麦 / 多人房 / 大房间**：**SFU**（各发一路、各选订阅）/ **MCU**（云端合成）**概念上提一句**即可，不必展开数学。

---

## 5. 音频：3A 与「典型现象」

| 能力 | 英文缩写 | 面试常挂的现象 |
|------|----------|----------------|
| 回声消除 | AEC | 开外放、双方同时说话、硬件差 → **啸叫/回音** |
| 噪声抑制 | ANS | 底噪、地铁风噪；过度 → **语音发飘、吞字** |
| 自动增益 | AGC | 远讲近讲、忽大忽小 |

**可说的话**：  
「3A 一般在 SDK 里默认开；**业务侧**更多遇到的是 **audio session category/mode 配置** 和 **外放/蓝牙切换** 导致的问题，需要结合系统日志和厂商建议排查。」

---

## 6. iOS 侧高频踩坑（与「懂不懂 SDK 内部」弱相关，但很加分）

- **AVAudioSession**：`PlayAndRecord`、是否 **默认外放/听筒**、和 **系统电话/VoIP/蓝牙** 的抢占与恢复。  
- **App 状态**：`UIApplication` 前后台、**音频后台模式** 声明；退后台时 **采集是否停**、**重连** 是否触发。  
- **多实例**：单例 **RTC Engine** 与多房间（若业务需要）的 **互斥/复用**；**泄漏** 导致第二次进房异常。  
- **线程与回调**：**厂商回调在何线程**、与主线程 UI/业务锁 **死锁/时序** 问题。  

---

## 7. 弱网、质量与「能说的指标」

- **RTT/丢包/抖动**：面试不需要精确公式，能说「**高 RTT 拉长全链路、丢包要重传或 FEC、抖动大靠 buffer**」即可。  
- **首包 / 首帧 / 可听时延**：可对应自埋点里 **进房到「听见对方」** 的耗时；与 **DNS、信令、媒体建连、订阅第一帧** 分段。  
- **观感和埋点对照**：**卡顿/断续/机械音/飘音** 往往要分是 **网络** 还是 **设备/3A/蓝牙**。  

**若简历可量化**：补 **进房成功率、首帧 p50/p90、重连率、某错误码趋势**，比空泛说「很流畅」更硬。

---

## 8. 排障：从现象到「分层怀疑」

1. **只有我一个人有问题** → 本机 **权限/路由/蓝牙/双 App**、**版本/SDK 参数**。  
2. **某一类机型多** → **系统版本、机型的音频 bug、厂商已知问题**。  
3. **全平台同时炸** → **网络、机房、或业务服务**；先看 **信令/HTTP** 与 **媒体错误** 是否同向。  
4. **时好时坏** → **弱网/抖动/服务端负载**、**JitterBuffer** 与策略。

**和厂商配合**：**最小可复现 Demo、复现时间窗、** `日志 + 进房 ID + 双方 UserId`、**升级/回滚记录**；避免只扔一句「有杂音」。

---

## 9. 常见面试追问 — 超短口播

**Q：你们用的是 Web RTC 还是自研？**  
A：我们集成 **某厂商云 RTC**；**标准 WebRTC 是开源框架**，但商业 SDK 会在 **信令、弱网、传输与运维** 上包一层。业务侧不直接改 **SDP/ICE** 的话，**理解链路即可**。

**Q：延遲能优化到多少？**  
A：语音常见 **百毫秒到数百毫秒** 量级，受 **端上缓冲、网络、是否转码/混流、订阅策略** 影响；**可答自家埋点区间**，并说明 **优化手段在「减少不必要缓冲、网络选路、订阅路数」**，而非拍脑袋 50ms。

**Q：你做过性能优化吗？**  
A：可答 **路数/订阅、码率档、合流策略、前后台行为、与厂商版本升级**；iOS 上还可提 **CPU/发热、音频线程、日志级别** 等**工程面**。

**Q：如果你来我们会做自研，你怎么看？**  
A：**业务早期与多数场景，商业 RTC 是合理选择**；自研要评估 **编解码/传输/全球节点/合规** 等成本。个人可补 **系统与网络基础**，在团队需要时再深入**指定模块**。

## 10. 延伸自学（按投入排序）

1. 官方文档：**你用过的一家** RTC 文档里的 **架构图、关键概念、错误码、最佳实践**（性价比最高）。  
2. 概念书/专栏：**《WebRTC 技术详解》** 类或文章：SDP、ICE、STUN/TURN 做 **知道名词** 即可。  
3. 开源对照：**WebRTC 源码** 仅当岗位明确要求再啃。
